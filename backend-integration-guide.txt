# Comprehensive Backend Integration Guide

This document outlines the necessary steps and backend API design required to connect your Node.js server with this pre-configured Next.js frontend.

## 1. Environment Variable Setup

The first and most crucial step is to tell the Next.js application where your backend server is located.

-   **File**: `.env`
-   **Variable**: `NEXT_PUBLIC_API_BASE_URL`
-   **Action**: Update this variable with the full URL of your deployed or local backend server.
    -   Example for local development: `NEXT_PUBLIC_API_BASE_URL=http://localhost:3001`
    -   Example for production: `NEXT_PUBLIC_API_BASE_URL=https://your-api.yourdomain.com`
    
    _Note: The frontend will automatically prefix this with `/api`_

## 2. API Endpoint Implementation

Your Node.js backend needs to implement the following endpoints. The frontend is already set up to call them.

### Authentication Endpoints

**Technology:** Use **JWT (JSON Web Tokens)** for authentication. Upon successful login, the backend should generate a JWT and send it back to the client. The client will then include this token in the `Authorization` header for all subsequent authenticated requests (e.g., `Authorization: Bearer <token>`).

**Password Hashing:** Use **bcrypt** to hash user passwords before storing them in the database.

**A. Register New User**
-   **Endpoint**: `POST /api/auth/register`
-   **Request Body (JSON)**:
    ```json
    {
      "name": "String",
      "email": "String",
      "password": "String"
    }
    ```
-   **Backend Logic**: Validate input, check for existing user, hash password, create new user.
-   **Success Response (201 Created)**:
    ```json
    { "message": "User registered successfully" }
    ```

**B. Login User**
-   **Endpoint**: `POST /api/auth/login`
-   **Request Body (JSON)**:
    ```json
    {
      "email": "String",
      "password": "String"
    }
    ```
-   **Backend Logic**: Find user by email, compare password hash, generate JWT.
-   **Success Response (200 OK)**:
    ```json
    { "token": "your_generated_jwt_string" }
    ```

**C. Get User Profile (Authenticated)**
-   **Endpoint**: `GET /api/auth/profile`
-   **Headers**: `Authorization: Bearer <token>`
-   **Backend Logic**: Verify JWT, fetch user data from the database.
-   **Success Response (200 OK)**:
    ```json
    {
      "id": "String",
      "name": "String",
      "email": "String"
    }
    ```

### Build Endpoints

**A. Create a New Build (Authenticated)**
-   **Endpoint**: `POST /api/builds`
-   **Headers**: `Authorization: Bearer <token>`
-   **Request Body (JSON)**:
    ```json
    {
      "carModel": "String",
      "color": "String",
      "selectedParts": ["String"]
    }
    ```
-   **Success Response (201 Created)**:
    ```json
    { "message": "Build saved successfully", "buildId": "new_build_id" }
    ```

**B. Get All User Builds (Authenticated)**
-   **Endpoint**: `GET /api/builds`
-   **Headers**: `Authorization: Bearer <token>`
-   **Success Response (200 OK)**:
    ```json
    [
      {
        "id": "String",
        "userId": "String",
        "carModel": "String",
        "color": "String",
        "selectedParts": ["String"],
        "createdAt": "ISOStringDate"

    ```

**C. Get a Specific Build (Authenticated)**
-   **Endpoint**: `GET /api/builds/:id`
-   **Headers**: `Authorization: Bearer <token>`
-   **Success Response (200 OK)**: (Same structure as GET /api/builds)

**D. Update a Build (Authenticated)**

-   **Headers**: `Authorization: Bearer <token>`
-   **Request Body (JSON)**: (Same as POST /api/builds)
-   **Success Response (200 OK)**:
    ```json
    { "message": "Build updated successfully" }
    ```

**E. Delete a Build (Authenticated)**
-   **Endpoint**: `DELETE /api/builds/:id`
-   **Headers**: `Authorization: Bearer <token>`
-   **Success Response (200 OK)**:
    ```json
    { "message": "Build deleted successfully" }
    ```

## 3. Database Collections/Tables Schema

Your MongoDB collections or SQL tables should be structured as follows.

**`users`**
```
{
  id: String (UUID),
  name: String,
  email: String,        // Should be unique
  passwordHash: String, // The bcrypt hashed password
}
```

**`services`**
```
{
  id: String (UUID),
  name: String,
  description: String,
  price: String         // e.g., "Starting from $500"
}
```

**`builds`**
```
{
  id: String (UUID),
  userId: String,     // Foreign key to the 'users' collection/table
  carModel: String,
  color: String,
  selectedParts: [String], // Array of selected part names, e.g. ["wheels", "spoiler"]
  createdAt: Date
}
```

**`user_vehicles`**
```
{
  id: String (UUID),
  userId: String,     // Foreign key to the 'users' collection/table
  make: String,
  carModel: String,
  color: String,
  year: Number,
  odoRead: String     // e.g., "15,000 Km"
}
```

## 4. Hardcoded Data

For development and demonstration purposes, some data collections are currently hardcoded in the frontend (`src/lib/constants.ts`) and loaded via a state manager (`src/store/data-store.ts`). In a production application, you should replace the mock data fetching logic with actual API calls to your backend endpoints.

-   **Hot Collections & Regular Collections**: Fetch from `/api/vehicles` or a dedicated `/api/cars-for-sale` endpoint.
-   **Active Services / Ongoing Service**: This would likely be derived from a `user_services` or similar collection that tracks services booked by users.
-   **Available Services**: Fetch from `/api/services`.
